## Model Architecture

`backend/apps/careers/models.py` defines a single `Job` model extending `TimeStampedModel`.

| Area | Fields |
|------|--------|
| Identity | `title_en`, `title_ar`, `slug` (unique) |
| Organization | `department_en`, `department_ar`, `location_en`, `location_ar` |
| Classification | `employment_type`, `workplace_type`, `experience_level` |
| Summary | `short_description_en`, `short_description_ar`, `description_en`, `description_ar` |
| Responsibilities | `responsibilities_en`, `responsibilities_ar` |
| Requirements | `requirements_en`, `requirements_ar` |
| Preferred qualifications | `preferred_qualifications_en`, `preferred_qualifications_ar` |
| Benefits | `benefits_en`, `benefits_ar` |
| Application | `application_method`, `external_apply_url`, `application_email` |
| Dates & display | `posted_date`, `closing_date`, `display_order` (db_index), `is_featured`, `is_active`, `show_on_homepage` |
| SEO | `seo_title_en`, `seo_title_ar`, `seo_description_en`, `seo_description_ar` |
| Timestamps | `created_at`, `updated_at` (from `TimeStampedModel`) |

No separate taxonomy models, no applicant model, no CV upload, no recruitment pipeline fields.

- **List display**: English title, Arabic title, department, location, employment type, workplace type, experience level, posted date, closing date, featured, homepage, active, updated timestamp.

## API Endpoints

| Method | URL | Description |
|--------|-----|-------------|
| GET | `/api/v1/jobs/` | Active, non-expired jobs; supports filters |
| GET | `/api/v1/jobs/<slug>/` | Single active, non-expired job by slug |

### Supported list filters

- `is_featured` — boolean (`true` / `false`)
- `show_on_homepage` — boolean (`true` / `false`)
- `employment_type`
- `workplace_type`
- `experience_level`
- `department` — case-insensitive match against English or Arabic department

Inactive and expired jobs are excluded from both endpoints. Unknown/inactive/expired slugs return `404`.

## Frontend Integration

### API layer

`src/services/careersApi.js` wraps `apiFetch`:

- `getJobs(filters, options)` → `GET /api/v1/jobs/?...`
- `getJob(slug, options)` → `GET /api/v1/jobs/<slug>/`

### Hook

`src/hooks/useJobs.js` uses the existing `useI18n` hook and `getBilingual` utility. It accepts optional filters, normalizes CMS payloads into language-resolved strings, and returns `{ jobs, loading, error }`. It returns `null` until valid data arrives so consumers can render their own fallback.

### CareersSection integration

`src/components/sections/CareersSection.jsx` now fetches `show_on_homepage=true` jobs via `useJobs`. The existing four hardcoded cards are rendered immediately as fallback and replaced after a successful fetch. Animations, delays, responsive behavior, and the contact CTA are preserved.

## Careers Page Behavior

`src/pages/CareersPage.jsx` at `/careers`:

- Lists active, non-expired jobs from the CMS.
- Falls back to the same four hardcoded career cards if the API fails.
- Displays title, department, location, employment type, workplace type, experience level, and short description.
- Provides an "Apply" button that:
  - opens external URLs in a new tab for `external_url`
  - uses `mailto:` for `email`
  - scrolls to the contact section (or navigates home then scrolls) for `contact_page`
- Supports bilingual rendering and RTL via the existing `useI18n` provider.
- Shows a professional empty state when CMS data is present but contains no records.

## Language Behavior

- `useJobs` consumes `lang` from the existing `I18nProvider`.
- `getBilingual` resolves CMS objects to the active language, falling back to English when Arabic is missing.
- `document.documentElement.dir` is already managed by `I18nProvider`; the careers page uses `dir={dir}` on its root element.
- New UI labels are translated in `src/i18n/en.js` and `src/i18n/ar.js`.

---

## Fallback Strategy

- `CareersSection` renders hardcoded cards immediately (`jobs` starts as `null`).
- On successful CMS fetch, the section re-renders with CMS cards.
- On failure, the hardcoded cards remain visible; the section does not crash.
- `CareersPage` uses the same pattern: fallback cards render when `jobs` is `null`.
- No intrusive loaders; loading text is shown only on the dedicated page and is subtle.

---

## Expiration Behavior

- `Job.public_qs()` returns active jobs with `closing_date` either `null` or `>= today`.
- Both list and detail views use this queryset.
- Expired jobs are excluded from public APIs but remain visible and editable in Django Admin.
- Jobs without a closing date are evergreen and remain public while active.

The model and API are designed for the future Sidrah-branded CMS dashboard. The authenticated operations it will need are:

- Reorder (edit `display_order`)

No custom write APIs or dashboards are implemented in this phase.

## Known Limitations

- The seeded records are evergreen career tracks, not concrete live vacancies. Titles, descriptions, and department values are derived directly from the existing `CareersSection` fallback content.
- The dedicated careers page does not include a job detail route; each card uses an apply action instead, matching the current interaction design.
- Header navigation is CMS-driven (`useHeaderNavigation`). Adding a visible careers link there requires updating the Navigation CMS entry or adding a hardcoded fallback entry; that change was out of scope for this task.
- Long-form content fields (`description`, `responsibilities`, etc.) are empty in seeded data because the original four cards only provided short descriptions.
